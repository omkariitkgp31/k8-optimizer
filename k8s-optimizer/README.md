## Overview

In Kubernetes environments, workloads are chronically overprovisioned, leading to unnecessary infrastructure costs and inefficient cluster utilization. This optimizer engine ingests workload metrics, detects overprovisioned or underprovisioned deployments, and generates safe, actionable resource recommendations.

Built with **FastAPI** and **Python 3.11**, it exposes a REST API for ad-hoc analysis and ships with native **Kubernetes manifests**, **Prometheus metrics**, and **Docker** support for immediate deployment.

## Features

| Feature | Description |
|---------|-------------|
| **Resource Analysis** | Compares requested vs. actual CPU and memory utilization |
| **Safe Recommendations** | Applies configurable safety buffers and hard floors to prevent OOMKill |
| **Noise Gate** | Suppresses marginal changes (< 15%) to avoid unnecessary pod churn |
| **Prometheus Metrics** | Exposes `recommendations_issued_total` counter with deployment/action labels, plus default HTTP latency/request metrics |
| **Environment Configuration** | All thresholds tunable via environment variables (12-Factor App compliant) |
| **Kubernetes-Native** | Includes Namespace, ConfigMap, Deployment, and Service manifests |
| **Health Probes** | Liveness and readiness endpoints for production orchestration |
| **Input Validation** | Strict Pydantic schemas with readable, field-level error messages |

## Architecture & Design Decisions

- **Units:** CPU is measured in **millicores** (1000m = 1 vCPU). Memory is measured in **MiB**.
- **Average-Based Analysis:** The current implementation uses average utilization. A **1.5× safety buffer** compensates for hidden spikes that averages cannot reveal. In production, this should be replaced with p95/p99 percentiles.
- **Memory Conservatism:** Memory uses wider "no-change" bands than CPU because **OOMKill is fatal**; CPU only throttles and is recoverable.
- **Churn Prevention:** A **15% noise gate** prevents the system from recommending pod restarts that save trivial amounts of resources.
- **Best-Effort Skipping:** Workloads with `cpu_request = 0` or `memory_request = 0` are treated as best-effort QoS and skipped silently.
- **Static Request Scope:** HPA/VPA interactions are out of scope for this version. The optimizer handles static resource requests only.

## Configuration

All optimizer thresholds are externalized via environment variables and injected through a Kubernetes ConfigMap. No code changes are required to tune operational behaviour.

| Variable | Default | Description |
|----------|---------|-------------|
| `CPU_HIGH_OVER_THRESH` | `0.35` | Below this CPU %, workload is highly overprovisioned |
| `CPU_MOD_OVER_THRESH` | `0.65` | Below this CPU %, workload is moderately overprovisioned |
| `CPU_UNDER_THRESH` | `0.85` | Above this CPU %, workload is underprovisioned |
| `MEM_HIGH_OVER_THRESH` | `0.40` | Below this memory %, highly overprovisioned |
| `MEM_MOD_OVER_THRESH` | `0.70` | Below this memory %, moderately overprovisioned |
| `MEM_UNDER_THRESH` | `0.85` | Above this memory %, underprovisioned |
| `CPU_HIGH_BUFFER` | `1.5` | Buffer multiplier for highly overprovisioned CPU |
| `CPU_MOD_BUFFER` | `1.3` | Buffer multiplier for moderately overprovisioned CPU |
| `CPU_UNDER_BUFFER` | `1.2` | Buffer multiplier for underprovisioned CPU |
| `MEM_HIGH_BUFFER` | `1.5` | Buffer multiplier for highly overprovisioned memory |
| `MEM_MOD_BUFFER` | `1.3` | Buffer multiplier for moderately overprovisioned memory |
| `MEM_UNDER_BUFFER` | `1.25` | Buffer multiplier for underprovisioned memory |
| `MIN_CPU` | `50` | Absolute floor for CPU recommendations (millicores) |
| `MIN_MEM` | `128` | Absolute floor for memory recommendations (MiB) |
| `CPU_ROUND` | `50` | Round CPU up to nearest multiple |
| `MEM_ROUND` | `128` | Round memory up to nearest multiple |
| `NOISE_GATE_PCT` | `0.15` | Ignore reductions smaller than this percentage |

## API Reference

### `POST /optimize`

Accepts a list of workload metrics and returns optimization recommendations.

**Request Body:**
```json
{
  "workloads": [
    {
      "deployment": "api-service",
      "cpu_request": 1000,
      "cpu_usage_avg": 180,
      "memory_request": 2048,
      "memory_usage_avg": 700
    },
    {
      "deployment": "worker-service",
      "cpu_request": 500,
      "cpu_usage_avg": 450,
      "memory_request": 1024,
      "memory_usage_avg": 900
    }
  ]
}
```

**Response (200 OK):**
```json
[
  {
    "deployment": "api-service",
    "recommended_cpu": 300,
    "recommended_memory": 1152,
    "reason": "CPU overprovisioned: usage is 18% of request; Memory overprovisioned: usage is 34% of request"
  }
]
```

### `GET /health`

Liveness and readiness probe endpoint.

**Response:**
```json
{
  "status": "healthy"
}
```

### `GET /metrics`

Prometheus-compatible telemetry endpoint exposing HTTP request metrics and custom business counters.

## Prometheus Metrics

The `/metrics` endpoint exposes the following time series:

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `http_request_duration_seconds` | Histogram | `handler`, `method`, `status` | Request latency |
| `http_requests_total` | Counter | `handler`, `method`, `status` | Request throughput |
| `http_requests_inprogress` | Gauge | `handler`, `method` | Concurrent requests |
| `recommendations_issued_total` | Counter | `deployment`, `resource_type`, `action` | Business metric tracking increase/decrease recommendations per deployment |

**Example scrape output:**
```
recommendations_issued_total{action="decrease",deployment="api-service",resource_type="cpu"} 1.0
recommendations_issued_total{action="decrease",deployment="api-service",resource_type="memory"} 1.0
```

## Discussion: Extending to Production Scale

To transition this proof-of-concept into a production-grade, large-scale Kubernetes optimizer, several architectural shifts are required:

### 1. Kubernetes APIs Integration
Instead of relying on manual JSON payloads, the system should operate as a **Custom Kubernetes Controller** or **Mutating Admission Webhook**. 
- It would watch the `metrics.k8s.io` API to fetch real-time utilization.
- It could create a CRD (Custom Resource Definition) like `OptimizationRecommendation` directly within the cluster so cluster admins can track recommendations natively using `kubectl`.
- Using a Mutating Webhook, it could auto-inject recommended resource requests/limits during pod creation (similar to how the Vertical Pod Autoscaler operates).

### 2. Metrics Collection
Relying on "average usage" is risky in production. 
- We must integrate directly with **Prometheus, Thanos, or Datadog** to run continuous PromQL queries targeting historical `p95` and `p99` percentiles over a 7-to-14 day window. 
- This protects against OOMKills during sudden traffic spikes, which averages fail to reveal. 
- The system should distinguish between CPU throttling metrics (e.g., `container_cpu_cfs_throttled_seconds_total`) and pure utilization to make safer CPU limit recommendations.

### 3. Scalability Considerations
In a cluster with 10,000+ pods, running optimization synchronously on a single API thread is unfeasible.
- We would implement a worker-queue architecture using **Kafka, RabbitMQ, or Celery**. A lightweight agent gathers pod usage, pushes events to a message broker, and horizontally-scaled workers process the rules engine asynchronously.
- Caching API Server responses (using SharedInformer caches natively in the Python/Go client) is critical to prevent the optimizer from overwhelming the Kubernetes API control plane.

### 4. Real-time Recommendations
Constantly updating pod resource requests requires restarting the pods, causing extreme churn and potential outages.
- Real-time "immediate" recommendations must be gated aggressively (e.g., only trigger an immediate emergency restart if memory usage is > 95% to prevent an imminent OOM crash).
- Standard optimizations should be batched and applied during scheduled maintenance windows, CI/CD deployment rollouts (e.g., GitOps updates via ArgoCD/Flux), or non-peak hours.

### 5. Multi-cluster Environments
For multi-cluster setups across different regions or cloud providers (AWS EKS, GCP GKE):
- A centralized control plane architecture is required.
- Lightweight agents deployed on edge/spoke clusters would forward sanitized metric aggregates to a central processing hub via gRPC.
- Recommendations must account for hardware differences (e.g., ARM vs x86 nodes) because CPU millicores behave differently across architectures.

### 6. Reliability Challenges
- **Interference with HPA/VPA:** Our optimizer could fatally conflict with the Horizontal Pod Autoscaler (HPA), which scales based on utilization percentages. If we blindly lower a pod's requested CPU, we artificially inflate its utilization percentage, which could trigger a massive, unwarranted HPA scale-out. The optimizer must be deeply "HPA-aware".
- **Fail-open design:** If the optimizer goes down, standard cluster scheduling and scaling must remain unaffected. It must never act as a single point of failure in the critical path of application deployments.

## Assumptions

1. All incoming CPU requests and usages are measured in **millicores** (1000m = 1 vCPU).
2. All incoming memory requests and usages are measured in **MiB**.
3. A **15% noise gate** is required to prevent Kubernetes API churn and unnecessary pod restarts.
4. The optimizer does not interact with Horizontal Pod Autoscaler (HPA) or Vertical Pod Autoscaler (VPA) targets in this version.
5. Workloads with `cpu_request = 0` or `memory_request = 0` are treated as best-effort and excluded from optimization.
